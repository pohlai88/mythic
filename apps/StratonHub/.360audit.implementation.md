### Findings (Good / Broken)

#### What’s Good (10 strengths)

1. **Clear persona split**: Strong audience gateways
   (`apps/StratonHub/app/page.tsx`,
   `apps/StratonHub/app/(audiences)/*/page.tsx`).
2. **Type-safe doc metadata**: Frontmatter schema is strict and canonical
   (`apps/StratonHub/lib/content/schemas.ts`).
3. **MDX posture consistency**: “Documentation Surface” posture is defined
   (`apps/StratonHub/docs/concepts/documentation-surface.md`) and used via
   `DocShell` layouts
   (`apps/StratonHub/app/(audiences)/developers/*/layout.tsx`).
4. **Search architecture exists**: Build-time index + runtime API route + cmd
   palette (`apps/StratonHub/lib/search/index-builder.ts`,
   `apps/StratonHub/app/api/search/route.ts`,
   `apps/StratonHub/components/docs/CommandPalette.tsx`).
5. **Reader preference approach is hydration-safe**: Data-attributes approach
   (`apps/StratonHub/components/docs/DocLayout.tsx`).
6. **Governance is unusually mature**: Intake + ratification + regression
   checklist + silence budget (`apps/StratonHub/docs/governance/*`).
7. **Explicit “Refuse List” discipline**: Prevents common docs UX bloat
   (`apps/StratonHub/docs/governance/REFUSE_LIST.md`).
8. **MDX typography mapping is centralized**: Headings/paragraphs styled
   consistently (`apps/StratonHub/mdx-components.tsx`).
9. **Doc templates exist**: Diataxis templates are ready-to-use
   (`apps/StratonHub/templates/content-schemas/*`).
10. **App config is production-minded**: Security headers + MDX setup
    (`apps/StratonHub/next.config.mjs`).

#### What’s Broken (10 failures)

1. **Major dead-ends in User hub**: `/users` advertises many surfaces but only
   `boardroom` (and `training`) exists as routes
   (`apps/StratonHub/app/(audiences)/users/page.tsx` vs directory
   `apps/StratonHub/app/(audiences)/users/`).
2. **Training lists modules that are not navigable**: Operator training
   enumerates surfaces without corresponding pages
   (`apps/StratonHub/app/(audiences)/users/training/page.mdx`).
3. **Search likely non-functional by default**: Search API loads
   `public/search-index.json` (`apps/StratonHub/app/api/search/route.ts`), but
   `apps/StratonHub/public/` currently contains only `robots.txt` (no
   `search-index.json`).
4. **ADR system is referenced but empty**: Ratification process mandates ADRs
   (`apps/StratonHub/docs/governance/UX_RATIFICATION_PROCESS.md`), but decisions
   dir reports “No approved ADRs yet”
   (`apps/StratonHub/docs/governance/decisions/README.md`); meanwhile developer
   architecture page lists ADR-001/2/3 as if they exist
   (`apps/StratonHub/app/(audiences)/developers/architecture/page.mdx`).
5. **Content bloat / paste-and-go symptoms**: “Coming soon” blocks and
   placeholder sections undermine “canonical” claim
   (`apps/StratonHub/app/(audiences)/developers/guides/page.mdx`).
6. **Template link targets are non-canonical**: Templates reference routes that
   don’t exist in this hub (e.g. `/how-to/...`, `/reference/...`)
   (`apps/StratonHub/templates/content-schemas/*.mdx`).
7. **Surface taxonomy drift**: Canonical surface list includes `global-config`
   and `individual-config` (`apps/StratonHub/lib/content/schemas.ts`), but
   `ModuleNav` and `/users` surfaces lists don’t
   (`apps/StratonHub/components/docs/ModuleNav.tsx`,
   `apps/StratonHub/app/(audiences)/users/page.tsx`).
8. **Business persona missing benchmark essentials**: No KPI dictionary,
   operating model, process flows, or decision log (only
   overview/compliance/training) (`apps/StratonHub/app/(audiences)/business/*`).
9. **Developer persona missing enterprise essentials**: No “data model/DB/RLS”,
   no contract schemas (Zod/OpenAPI), no runbooks/troubleshooting pages beyond
   README-level notes (`apps/StratonHub/app/(audiences)/developers/*`,
   `apps/StratonHub/README.md`).
10. **Label/intent mismatch in 404 wayfinding**: “Search Archives” CTA routes to
    `/developers` (not search) (`apps/StratonHub/app/not-found.tsx`).

### Cursor IDE Execution Rules

- **No hallucination**: Every claim must cite an existing file path; do not
  invent pages/routes (“dead-end” must be shown via path mismatch like
  `apps/StratonHub/app/(audiences)/users/page.tsx` vs missing folder).
- **Never invent “already exists” ADRs**: If an ADR is missing, propose a **new
  file skeleton** under `apps/StratonHub/docs/governance/decisions/` and label
  it as new work.
- **If missing content**: Propose **minimal skeletons only** (frontmatter + H1 +
  required headings). No filler prose.
- **DRY edits**: Prefer shared templates
  (`apps/StratonHub/templates/content-schemas/*`) and shared components
  (`apps/StratonHub/components/docs/*`) over duplicating patterns.
- **Paragraph discipline**: Max **6 lines per paragraph** in MD/MDX.
- **Heading hierarchy**: Strict **H1 → H2 → H3** (no skipping).
- **Diataxis purity**: One Diataxis mode per page (enforced by
  `apps/StratonHub/docs/governance/INTAKE_CHECKLIST.md` and schema in
  `apps/StratonHub/lib/content/schemas.ts`).
- **Search/index changes must be testable**: Any search change must account for
  `apps/StratonHub/app/api/search/route.ts` dependency on
  `public/search-index.json` and build pipeline in
  `apps/StratonHub/scripts/build-search-index.ts`.
